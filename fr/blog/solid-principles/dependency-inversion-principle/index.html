📖 Qu'est-ce que le principe d'inversion de dépendance ?

Le principe d'inversion de la dépendance dit :

Les modules de haut niveau ne doivent pas dépendre des modules de bas niveau. Les deux doivent dépendre d'abstractions. Les abstractions ne doivent pas dépendre des détails. Les détails doivent dépendre des abstractions.

En d'autres termes :
Dépendre des interfaces, pas des implémentations concrètes.

❌ Une violation typique du DIP

public class LightBulb {
    public void turnOn() { ... }
    public void turnOff() { ... }
}

public class Switch {
    private LightBulb bulb = new LightBulb();

    public void operate() {
        bulb.turnOn(); // tightly coupled
    }
}

Ici, Switch dépend directement de LightBulb, un module de bas niveau. Changer LightBulb signifie modifier Switch.

✅ Appliquer le DIP avec l'abstraction

Introduire une interface :

public interface Switchable {
    void turnOn();
    void turnOff();
}

Maintenant, découplez :

public class LightBulb implements Switchable {
    public void turnOn() { ... }
    public void turnOff() { ... }
}

public class Switch {
    private Switchable device;

    public Switch(Switchable device) {
        this.device = device;
    }

    public void operate() {
        device.turnOn();
    }
}

Maintenant, Switch dépend de l'abstraction Switchable, et non de la classe concrète.

🧪 Analogie du monde réel : Passerelle de paiement

Imaginez que votre application gère les paiements :

class PaymentService {
    private PayPalAPI paypal = new PayPalAPI();

    public void pay() {
        paypal.sendMoney();
    }
}

Demain, vous passez à Stripe. C'est un cauchemar.

Mieux:

interface PaymentGateway {
    void pay();
}

class PayPalGateway implements PaymentGateway { ... }
class StripeGateway implements PaymentGateway { ... }

class PaymentService {
    private PaymentGateway gateway;

    public PaymentService(PaymentGateway gateway) {
        this.gateway = gateway;
    }

    public void processPayment() {
        gateway.pay();
    }
}

Vous pouvez désormais changer de prestataire de paiement sans réécrire la logique métier.

🧠 Avantages du DIP

  • 🚦 Couplage souple
  • 🔄 Facilité de modification des implémentations
  • 🧪 Meilleure testabilité (mocks, stubs)
  • 🔧 Architecture plus propre

📘 Recap

  • DIP = S'appuyer sur des abstractions et non sur des implémentations concrètes
  • Les modules de haut niveau et de bas niveau doivent dépendre d'interfaces
  • Permet la flexibilité, la testabilité et la maintenabilité à long terme
