Ajouter la couche validation

Dans la première version du projet, il n'y a aucune validation est faite aux APIs de création et de modification de Todo et on peut se retrouver avec des tâches Todo sans titre et description stockées en base de données. Pour pallier à ce problème, nous allons créer le package validator dans lequel nous allons ajouter l'interface ITodoValidator et son implémentation TodoValidator qui va nous permettre de valider une tâche Todo:

  • TodoValidator

    package net.learntime.todo.api.validator;

import net.learntime.todo.api.exception.TodoBadRequestException;
import net.learntime.todo.api.model.Todo;
import org.apache.commons.lang3.StringUtils;
import org.springframework.stereotype.Service;

import java.util.Objects;

@Service
public class TodoValidator implements ITodoValidator {

    public void validate(Todo todo) {
        if (StringUtils.isBlank(todo.getTitle())) {
            throw new TodoBadRequestException("title is mandatory");
        }
        if (StringUtils.isBlank(todo.getDescription())) {
            throw new TodoBadRequestException("description is mandatory");
        }
        if (Objects.isNull(todo.getCompleted())) {
            throw new TodoBadRequestException("completed is mandatory");
        }
    }
}

Dans la classe TodoValidator, nous avons utilisé la classe StringUtils que nous avons importé de la bibliothèque commons-lang3 de Apache et voici la modification que nous avons apporté au fichier pom.xml de notre projet pour pouvoir l'utiliser

<dependency>
	<groupId>org.apache.commons</groupId>
	<artifactId>commons-lang3</artifactId>
</dependency>

alert-info

La version de cette dependance a été héritée puisqu'il s'agit d'un projet Spring Boot

On a aussi créé une nouvelle exception TodoBadRequestException dans le package exception qu'on a levé une fois qu'on a une erreur de validation

package net.learntime.todo.api.exception;

public class TodoBadRequestException extends RuntimeException {

    public TodoBadRequestException(String message) {
        super(message);
    }

}

Pour appliquer cette validation, nous allons utiliser notre validator dans notre controller:

@RestController
@RequestMapping(path = "/todos")
public class TodoController {

    @Autowired
    private ITodoValidator validator;

    @PostMapping
    public Todo create(@RequestBody Todo todo) {
        validator.validate(todo);
        return repository.save(todo);
    }

    @PutMapping("{id}")
    public Todo update(@PathVariable Long id, @RequestBody Todo todo) {
        validator.validate(todo);
        var entity = repository.findById(id).orElseThrow(() -> new TodoNotFoundException(id));
        entity.setTitle(todo.getTitle());
        entity.setDescription(todo.getDescription());
        entity.setCompleted(todo.getCompleted());
        repository.save(entity);
        return entity;
    }
}

Avec cette couche de validation, on obient une erreur quand on essaye de créer ou mettre à jour une tâche Todo invalide

Conclusion

Nous avons maintenant intégré une couche de validation dans notre API REST pour garantir que chaque tâche Todo possède un titre, une description et un état de complétion valides.

Cependant, nous avons constaté que notre API renvoie actuellement un statut d'erreur 500 Internal Server Error lors de la validation, alors que le comportement attendu est un statut 400 Bad Request avec un message d'erreur informatif.

Dans la prochaine section, nous allons mettre en place les Exceptions Handlers pour pouvoir modifier le statut http et le body de la réponse retournée en cas d'erreur.
