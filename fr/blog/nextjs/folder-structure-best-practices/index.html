Next.js Structure du dossier Meilleures pratiques

Next.js a évolué vers un framework full-stack, ce qui signifie que vous écrivez du code qui s'exécute à la fois sur le serveur (Node.js) et sur le client (navigateur). C'est puissant, mais cela signifie aussi que votre projet peut rapidement devenir désordonné si vous mélangez les préoccupations.

Dans cet article, nous allons voir comment structurer votre projet pour que votre code soit bien organisé, sécurisé et évolutif - sans ralentir le développement.

L'importance de la structure

Lorsque votre structure de dossiers se développe organiquement sans directives, vous prenez des risques :

  • Fuites de sécurité - transmission accidentelle de secrets au navigateur.
  • Problèmes de performance** - importation de la logique lourde du backend dans les bundles du frontend.
  • Confusion des développeurs - personne ne sait où se trouvent les fichiers.

Une bonne structure :

  • Empêche les importations accidentelles d'un environnement à l'autre.
  • La taille des paquets reste réduite.
  • Rend votre base de code prévisible pour les nouveaux développeurs.

Les trois types de code dans Next.js

Lorsque vous pensez à l'endroit où le code s'exécute dans un projet Next.js, vous pouvez le regrouper en trois catégories :

1. Code réservé au serveur

Fonctionne uniquement en Node.js (jamais livré au navigateur).
Utilisez-le pour :

  • Requêtes de base de données
  • Logique d'authentification
  • Accès au système de fichiers
  • Envoi de courriels

✅ Importation sûre de bibliothèques réservées aux serveurs (par exemple, Prisma, AWS SDK).

2. Code réservé au client

Fonctionne uniquement dans le navigateur.
Utilisez-le pour :

  • Interactivité (useState, useEffect, gestionnaires d'événements)
  • API DOM (window, document)
  • Stockage local

✅ Doit inclure "use client" en haut de l'App Router.

3. Code partagé (isomorphe)

Fonctionne dans les deux environnements.
Utilisez-le pour :

  • Types TypeScript
  • Formatage de la date
  • Schémas de validation
  • Composants d'interface utilisateur purs (pas d'API de serveur, pas d'API de navigateur)

Une structure de dossiers pratique

Voici une façon simple d'organiser votre application Next.js pour que chaque type de code soit facile à trouver :

src/
├── server/                    # Runs only on the server
│   ├── actions/               # Server Actions
│   ├── db/                    # Database access
│   ├── services/              # Business logic
│   └── utils/                 # Server helpers (email, auth)
│
├── client/                    # Runs only in the browser
│   ├── components/            # Client UI components ("use client")
│   ├── hooks/                 # Client hooks
│   └── utils/                 # Browser-only helpers
│
├── shared/                    # Runs in both environments
│   ├── components/            # Pure components
│   ├── types/                 # Type definitions
│   └── utils/                 # Environment-agnostic helpers
│
├── app/                       # Next.js App Router entrypoints
└── styles/

Règles d'importation

  • server/ → peut importer de shared/.
  • client/ → peut importer de shared/.
  • ❌ client/ → cannot import from server/
  • ❌ server/ → ne devrait pas importer de client/.

Vous pouvez faire respecter cette règle par des règles ESLint ou simplement par la discipline des développeurs.

What Goes in shared/

Le dossier shared/ existe pour éviter la duplication du code qui fonctionne aux deux endroits :

Catégorie Exemple
Types types/user.ts
Constantes constants/roles.ts
Fonctions pures utils/formatDate.ts
Validation validation/userSchema.ts
Primitives d'interface utilisateur components/Text.tsx

Conseils de pro pour une séparation en toute sécurité

Marquer les fichiers réservés au serveur

import 'server-only'; // Throws error if imported into client code

Marquer les composants du client

"use client";

Utiliser ESLint pour bloquer les importations erronées

Configurez no-restricted-imports pour server/ → client/.

Réflexions finales

Séparer votre code en fonction de l'endroit où il s'exécute n'est pas seulement une question d'ordre - c'est aussi une question de sécurité, performance et bonheur des développeurs. Une fois que votre équipe suit une structure claire, vous passerez moins de temps à déboguer et plus de temps à créer des fonctionnalités.

L'état d'esprit clé :

Serveur = logique secrète, lourde, backend Client = interface utilisateur interactive Shared = code pur, agnostique de l'environnement

En respectant ces limites, votre application Next.js restera saine au fur et à mesure de sa croissance.
