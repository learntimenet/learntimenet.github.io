Introduction

Why Component Organization Matters

A good folder and file structure:

  • Improves readability ‚Äî anyone can navigate your code quickly.
  • Boosts reusability ‚Äî UI components can be shared across features.
  • Simplifies maintenance ‚Äî changes in one place update everywhere.

Recommended Folder Structure

Your folder structure should make it obvious where a component lives and why. With shadcn/ui and TailwindCSS, the goal is to separate base UI, common reusable parts, and feature-specific logic.

1. By Type (works well for small to medium apps)

src/
  components/
    ui/             # shadcn-generated base components (pure UI, no logic)
    common/         # wrappers, shared styled components
    layout/         # navigation bars, sidebars, footers
    features/       # feature-specific UI pieces
  hooks/            # custom React hooks
  lib/              # utilities, helper functions
  pages/            # page-level components (Next.js, Remix, etc.)
  styles/           # Tailwind globals and overrides

When to use:

  • ‚úÖ You‚Äôre building a small to mid-sized app
  • ‚úÖ You want to keep all UI under components/
  • ‚úÖ You don‚Äôt have a huge number of features yet

2. By Feature (best for large, complex projects)

src/
  features/
    auth/
      components/
        LoginForm.tsx
        SignupForm.tsx
      hooks/
        useLogin.ts
      services/
        authApi.ts
    dashboard/
      components/
      hooks/
      utils/
  components/
    ui/             # shadcn-generated
    common/         # app-wide shared UI

When to use:

  • ‚úÖ Your app has many distinct features
  • ‚úÖ Teams are working on different features in parallel
  • ‚úÖ You want to avoid scrolling through hundreds of components in one folder

Tips:

  • Avoid abbreviations unless widely understood.

The Role of shadcn/ui

The ui/ folder holds all components generated by the shadcn CLI (e.g., buttons, inputs, modals).
These are base components ‚Äî minimal styling, no business logic.

The common/ folder wraps these base components with project-specific styles or variants.
This allows:

  • Consistent branding
  • Centralized style updates
  • Safe shadcn updates without overwriting customizations

Example wrapper:

// components/common/PrimaryButton.tsx
import { Button } from "@/components/ui/button"

export function PrimaryButton(props) {
  return (
    <Button className="bg-blue-600 hover:bg-blue-700 text-white" {...props} />
  )
}

Styling with cva and cn

cva ‚Äî Class Variance Authority

Use cva to define style variants for reusable components:

import { cva } from "class-variance-authority"

const badge = cva("px-2 py-1 rounded text-sm", {
  variants: {
    color: {
      info: "bg-blue-100 text-blue-800",
      danger: "bg-red-100 text-red-800"
    },
    size: {
      sm: "text-xs",
      md: "text-sm"
    }
  },
  defaultVariants: {
    color: "info",
    size: "md"
  }
})

export function Badge({ children, color, size }) {
  return <span className={badge({ color, size })}>{children}</span>
}

cn ‚Äî Conditional Class Merging

Merge Tailwind classes dynamically:

import { cn } from "@/lib/utils"

function Alert({ type, children }) {
  return (
    <div
      className={cn(
        "p-4 rounded",
        type === "error" && "bg-red-100 text-red-800",
        type === "success" && "bg-green-100 text-green-800"
      )}
    >
      {children}
    </div>
  )
}

Using index.ts for Cleaner Imports

index.ts files act as ‚Äúbarrels‚Äù to re-export components, reducing import paths.

Instead of:

import { PrimaryButton } from "@/components/common/PrimaryButton"
import { SectionTitle } from "@/components/common/SectionTitle"

Use an index file:

// components/common/index.ts
export { PrimaryButton } from "./PrimaryButton"
export { SectionTitle } from "./SectionTitle"

Now you can import:

import { PrimaryButton, SectionTitle } from "@/components/common"

Best Practices Checklist

  • Keep base UI in ui/ untouched for easier updates.
  • Wrap base components in common/ for custom styles.
  • Group related components in feature folders.
  • Use cva for style variants, cn for conditional classes.
  • Use index.ts to simplify and centralize exports.
  • Define colors, spacing, and typography in tailwind.config.js.

Conclusion

A clean component organization strategy with shadcn/ui and TailwindCSS keeps your React project scalable, maintainable, and easy to navigate. By separating base UI from custom wrappers, grouping by feature, and using styling utilities, you ensure that your codebase remains consistent and easy to refactor ‚Äî no matter how big the app grows.
