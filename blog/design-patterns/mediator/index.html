üîç What Is the Mediator Pattern?

The Mediator Pattern defines an object that centralizes communication between other objects. Instead of objects referring to each other directly, they interact through the mediator.

Imagine an air traffic control tower: it coordinates planes instead of planes communicating directly with each other.

üß© When to Use It

You should use the Mediator Pattern when:

  • Objects are too tightly coupled and you want to reduce direct dependencies
  • You have complex communication between many classes
  • You want to centralize control and communication logic

üë©‚Äçüè´ Step 1: Define the Mediator Interface

public interface ChatMediator {
    void sendMessage(String message, User user);
    void addUser(User user);
}

üß± Step 2: Concrete Mediator

import java.util.ArrayList;
import java.util.List;

public class ChatRoomMediator implements ChatMediator {
    private List<User> users = new ArrayList<>();

    public void addUser(User user) {
        users.add(user);
    }

    public void sendMessage(String message, User sender) {
        for (User user : users) {
            // Don't send message to the sender
            if (user != sender) {
                user.receive(message);
            }
        }
    }
}

üë§ Step 3: Define the User Abstract Class

public abstract class User {
    protected ChatMediator mediator;
    protected String name;

    public User(ChatMediator mediator, String name) {
        this.mediator = mediator;
        this.name = name;
    }

    public abstract void send(String message);
    public abstract void receive(String message);
}

üôã‚Äç‚ôÇÔ∏è Step 4: Concrete User Implementation

public class ChatUser extends User {

    public ChatUser(ChatMediator mediator, String name) {
        super(mediator, name);
    }

    public void send(String message) {
        System.out.println(this.name + " sends: " + message);
        mediator.sendMessage(message, this);
    }

    public void receive(String message) {
        System.out.println(this.name + " receives: " + message);
    }
}

public class MediatorPatternDemo {

üåê Real-World Analogy

  • Air traffic control: Pilots communicate with a control tower instead of directly with each other.
  • Group chat apps: Messages are routed through a central server.
  • Smart home systems: Devices report to a hub rather than directly to each other.

‚ö†Ô∏è Pitfalls to Avoid

  • ‚ùå Overloading the mediator: it can become too complex if it handles too much logic
  • ‚ùå Creating unnecessary mediators for simple communications
  • ‚ùå Tight coupling to a specific mediator instead of using an interface

‚úÖ Best Practices

  • Keep the mediator focused and simple
  • Abstract the mediator through an interface for flexibility
  • Refactor communication logic gradually as systems become more complex

üìò Recap

  • Mediator Pattern centralizes communication to reduce dependencies between classes
  • It promotes loose coupling and improves maintainability
  • Best suited for chat rooms, air traffic control, and smart systems
